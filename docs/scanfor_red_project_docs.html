1. Project Overview

What Problem Does This Solve?

The IoTGW (Internet of Things Gateway) platform runs on dozens of servers across multiple environments. Every 10 minutes, a monitoring tool called Scanfor scans application log files on each server and counts how many times specific error patterns appear. These counts are written to .prom files (Prometheus-format metric files).

Without this dashboard, an on-call engineer would have to manually SSH into servers, read raw log files or metric files, and decide whether each error is new, expected, getting worse, or has gone away. At scale вЂ” dozens of servers, hundreds of potential error types вЂ” this is slow, error-prone, and exhausting.

The Solution

Scanfor Red automatically reads the .prom files, compares them to a catalog of Known Issues, classifies every alert into one of four categories, and presents everything in a web dashboard that any engineer can read at a glance. No SSH required.

The Four Alert Categories

CategoryBadgeMeaningAction Required
New / UnknownNEWError pattern has no match in the Known Issues catalogInvestigate immediately
KnownKNOWNMatches a known issue and count/growth is within normal rangeMonitor per runbook
WorseningWORSENINGMatches a known issue but count or growth exceeds the normal thresholdEscalate вЂ” runbook first step
ResolvedRESOLVEDWas present in the previous snapshot but is gone nowVerify, then archive

3. End-to-End Data Flow

Step-by-step: what happens every 10 minutes

  1. Scanfor writes .prom files. The Scanfor tool (already running on the production server) scans log files and writes one .prom file per host into /home/plmon/status/scanfor/. Each file contains rows of error counts for that host. Files are completely overwritten вЂ” not appended.
  2. Cron job fires. The OS cron scheduler runs python3 -m backend.cron_ingest. This is the only process that reads the .prom files. It runs independently of the FastAPI web server.
  3. Folder hash check. The ingestor computes a SHA-256 hash of all .prom file contents combined. If the hash matches the last successful snapshot in the database, the run exits immediately вЂ” nothing is written. This prevents duplicate batches when files haven't changed.
  4. Parsing. Each .prom file is parsed line by line. Comment lines (starting with #) are skipped. Data lines are parsed into structured records: hostname, filename, error_type, count, plus metadata labels.
  5. Growth calculation. For each parsed alert, the ingestor looks up the most recent snapshot in the database that has an alert with the same fingerprint. growth = current_count в€’ previous_count. On the very first run, previous count is 0, so growth equals the current count.
  6. Classification. Each alert is passed through the classifier along with the full Known Issues catalog fetched from the database. The classifier returns a category (new/known/worsening) based on fingerprint matching and threshold comparison.
  7. Resolution detection. Any alert that was active in the previous snapshot but is absent in the current snapshot is automatically marked resolved and written to the new batch.
  8. Database write. One AlertBatch row and N AlertEvent rows are committed to SQLite. A PromSnapshot row records the file hash and metadata for idempotency.
  9. Dashboard refresh. The next time any user clicks Refresh Data or reloads the page, the browser fetches from /api/alerts, which reads the latest batch from the database and returns the newly classified alerts.

4. Understanding .prom Files

What is a .prom file?

A .prom file is a text file that follows the Prometheus exposition format вЂ” a standard way to expose numeric metrics. Each line describes one metric measurement. You do not need to know anything about Prometheus to understand this project; the format is just structured text.

File Naming

One file per server. The filename identifies which server produced it: 

cars-cars1-plmon-scanfor.prom
chne-vm1-plmon-scanfor.prom
cbmw-bmw1-plmon-scanfor.prom

File Contents вЂ” Anatomy of a .prom file

# HELP scanfor_errors Total errors matching scanfor patterns
# TYPE scanfor_errors gauge
# Generated: 2026-07-01 08:55:37
# State file: /tmp/scanfor_state_cars_cars1_2026-06-30.dat
scanfor_errors{system_type="IotGW",system="cars",hostname="cars-cars1-plmon-scanfor",
               tenant="cars",filename="commandlistener-main.20260630",
               error_type="SQLException",error_index="11",color="red",
               known_error="false",note="",caused_by="TelemetryDao.java:231"} 13

Breaking it down field by field

FieldExampleWhat It Means
system_typeIotGWType of platform вЂ” always IotGW in this project
systemcarsTenant system code (cars, chne, cbmw, etc.)

The count value is the total since the state file was created, not since the last 10-minute window. Growth (the increase since the last scan) is computed by the ingestor by comparing to the previous snapshot in the database.

Resolution Logic

After processing current alerts, the ingestor compares the new set of fingerprints against the previous snapshot. Any alert that was active before but is missing now gets a new resolved event written with count = 0.

Note on Worsening vs. Known

A high count with zero growth does not classify as worsening вЂ” it stays as "known." Worsening requires actual upward movement (growth > 0). This prevents stable-but-high errors from flooding the worsening section.

6. Fingerprinting вЂ” How Alerts Match Known Issues

A fingerprint is a short string that uniquely identifies an alert pattern independent of daily dates and server node numbers. It allows the same recurring error to be matched to the same Known Issue every day across any server in a host group.

Fingerprint Format

env | hostPrefix | logScope | errorType

Examples:
  prod | mxqrpiog | listener-main | SQLException
  prod | ccgw-eastus2-prod | *server-main | javax_net_ssl

Normalization Rules

InputAfter NormalizationRule
cars-cars1-plmon-scanforcars-cars1-plmon-scanforHost prefix = strip trailing digit run
mxqrpiog02mxqrpiogTrailing "02" stripped
ccgw-eastus2-prod-ford-vm-01ccgw-eastus2-prod-ford-vm-01 suffix stripped
listener-main.20260630listener-mainDate suffix .YYYYMMDD stripped
hostname with "staging"env = stagingEnv extracted from hostname keywords

The Known Issues catalog stores fingerprint patterns with wildcards (e.g., *server-main matches commandserver-main, authserver-main, etc.). The classifier first tries exact fingerprint matching, then falls back to scope-based matching (host_scope, log_scope, error_type fields on the Known Issue record).

8. The Database (SQLite + SQLAlchemy)

What is SQLite?

SQLite is a database engine that stores everything in one file: backend/database/scanfor_red.db. There is no separate database server to install, configure, or connect to. The Python process reads and writes the file directly. It is ideal for single-server internal tools like this dashboard.

What is SQLAlchemy?

SQLAlchemy is the Python library used to talk to the database. Instead of writing raw SQL queries like SELECT * FROM alert_events, you write Python like db.query(AlertEvent).filter(...).all(). SQLAlchemy translates this to SQL. The table definitions are in backend/database/models.py.

12. Security Considerations

This is an Internal Tool

Scanfor Red is designed for use on an internal network or VPN. It does not have user authentication. Do not expose port 9000 directly to the public internet.

What Data Is Stored?

The SQLite database contains:

  • Error type names and counts (e.g., SQLException: 144)
  • Hostnames of production servers
  • Log file names
  • Triage notes entered by engineers

It does not contain: customer data, PII, passwords, secrets, API keys, or the actual contents of log files. The .prom files themselves only contain aggregate counts, not log content.

Network Exposure

ConcernCurrent BehaviorRecommendation
Port exposureFastAPI listens on 0.0.0.0:9000 when started with --host 0.0.0.0Bind to 127.0.0.1 and put Nginx in front if wider access is needed. Use VPN or firewall rules.
AuthenticationNoneAdd HTTP Basic Auth via Nginx or a FastAPI middleware if the network is not trusted.
CORSAllows all origins (allow_origin_regex=".*")Fine for internal tools. For production hardening, restrict to the specific server's IP/hostname.
SQL InjectionAll queries use SQLAlchemy ORM with parameterized queriesSafe вЂ” no raw SQL string concatenation.
Path traversalThe .prom folder path is set via environment variable, not user inputNever accept the .prom path from user input. Keep it in the environment config.

13. Developer FAQ

Q: Why SQLite instead of PostgreSQL or MySQL?

SQLite requires zero installation and configuration. The database is a single file on disk. For a single-server internal tool processing a few dozen alerts every 10 minutes, SQLite is completely appropriate and far simpler to operate. If the project grows to need multiple app servers sharing a database, migrating to PostgreSQL would be straightforward using the same SQLAlchemy models.

Q: Why does the backend serve the frontend? Isn't that wrong?

For a simple internal tool, having one process serve everything is fine. It eliminates CORS headaches (frontend and API are the same origin) and simplifies deployment (one process to start/stop). If you want to separate them later, remove the StaticFiles mount from main.py and deploy the frontend via Nginx.

Q: Why vanilla JS? No React/Vue?

No build pipeline means no npm, no Node.js version management, no webpack configuration. A developer can clone the repo, install Python dependencies with pip, and have the app running in under 5 minutes. The dashboard re-renders by rebuilding table HTML as strings via template literals вЂ” simple and debuggable in any browser's DevTools.

Q: What happens if the .prom folder is unavailable when cron runs?

The ingestor raises a FileNotFoundError, the cron script logs an error message and exits with code 2. The database is not modified. The dashboard continues serving the previous snapshot's data unchanged.

Q: What happens if two cron jobs run at the same time?

The folder hash check prevents duplicates. If the files haven't changed, the second run exits immediately with "skipped." If files changed between the two runs, both will attempt to write вЂ” SQLite's WAL mode handles concurrent writes safely for this workload.

Q: Can I add more .prom fields without breaking things?

Yes. The parser in prom_parser.py reads all label key-value pairs. Adding new labels to the .prom files will simply result in them being stored in the alert_events table columns that match (if the column exists) or silently ignored (if not). Add a new column to the model and a migration step if needed.

Q: How do I add a new Known Issue?

Two ways: (1) Click the "Known Issues" tab in the dashboard and use the Add button. (2) POST to /api/known-issues with the JSON body. The new KI will be used in classification on the next ingest run.

Q: The dashboard shows "mock data" warning. What's wrong?

The backend is not running. Start it with: python3 -m uvicorn backend.main:app --reload --port 9000 from the project root, then reload the page.

Q: How do I see what's in the database without a GUI tool?

Appendix A: Automated .prom Processing вЂ” Cron Job Setup

What This Does

This setup automates the "Process .prom Folder Now" button so that the dashboard automatically refreshes with new alert data every 10 minutes вЂ” exactly matching how often Scanfor writes new .prom files. Engineers can also still trigger a manual update at any time using the button or a single command.

Overview

The script backend/cron_ingest.py does all the work:

  • Reads all *.prom files from the configured folder
  • Computes growth by comparing to the previous snapshot
  • Classifies each alert (new / known / worsening / resolved)
  • Writes a new batch to the SQLite database
  • Skips silently if the files haven't changed since the last run

It runs independently of the FastAPI web server. Even if the web server is restarting or temporarily down, cron continues ingesting data.

Prerequisites

  • The app is checked out on the server at a known absolute path (e.g., /opt/scanfor-red)
  • Python 3.9+ is available at python3 (or use a full path like /usr/bin/python3)
  • Dependencies are installed: pip3 install -r requirements.txt
  • The database has been initialized: python3 -m backend.database.init_db
  • The .prom source folder exists: /home/plmon/status/scanfor

Step-by-Step Setup

  1. Find the full path to your project directory.
    You need the absolute path (not a relative one) for cron. 
    cd /path/to/Scanfor_Red_Email_Alerts_Dashboard
pwd
# Example output: /opt/scanfor-red
    Use whatever path pwd shows in all steps below.
  2. Test the script manually first.
    Run it exactly as cron will run it, using absolute paths and environment variables: 
    cd /opt/scanfor-red
SCANFOR_PROM_FILE_PATH=/home/plmon/status/scanfor python3 -m backend.cron_ingest
    You should see output like: 
    2026-07-01 09:10:00 INFO processed вЂ” files=18 metrics=71 new_events=71 resolved=0 batch=PROM-20260701-091000-45204f
    Or on a second run (when files haven't changed): 
    2026-07-01 09:10:05 INFO skipped вЂ” folder unchanged (hash=45204fc31036)
    If you see an error, fix it before continuing.
  3. Create the log file so cron can write to it. 
    sudo touch /var/log/scanfor_cron.log
sudo chmod 644 /var/log/scanfor_cron.log

# If you don't have sudo access, use a path in your home directory instead:
touch ~/logs/scanfor_cron.log
  4. Open the crontab editor. 
    crontab -e
    This opens your personal cron file in your system's default editor (usually vi or nano).

    To use nano instead of vi: 
    EDITOR=nano crontab -e
  5. Add the cron entry at the bottom of the file.
    Replace /opt/scanfor-red with your actual project path: 
    # Scanfor Red вЂ” process .prom folder every 10 minutes
*/10 * * * * cd /opt/scanfor-red && SCANFOR_PROM_FILE_PATH=/home/plmon/status/scanfor python3 -m backend.cron_ingest >> /var/log/scanfor_cron.log 2>&1
    crontab Format Reminder

    The five fields before the command are: minute hour day-of-month month day-of-week
    */10 * * * * = "every 10 minutes, every hour, every day"

    Save and exit the editor. In nano: Ctrl+O в†’ Enter в†’ Ctrl+X. In vi: :wq
  6. Verify the crontab was saved correctly. 
    crontab -l
    You should see your entry listed.
  7. Wait for the next 10-minute mark, then check the log. 
    tail -f /var/log/scanfor_cron.log
    You should see a new log line appear at :00, :10, :20, :30, :40, :50 of each hour. Press Ctrl+C to stop watching.
  8. Confirm data appeared in the dashboard.
    Open the dashboard in your browser and click Refresh Data. The batch timestamp at the top should update to within the last 10 minutes.

Triggering a Manual Update

You have three ways to trigger an immediate update outside of the cron schedule:

Option 1 вЂ” Dashboard Button (easiest)

Click the "Process .prom Folder Now" button in the top-right corner of the dashboard. The page will show a status message and auto-refresh the data.

Option 2 вЂ” Command line

cd /opt/scanfor-red
SCANFOR_PROM_FILE_PATH=/home/plmon/status/scanfor python3 -m backend.cron_ingest

Option 3 вЂ” API call (curl)

curl -s -X POST http://localhost:9000/api/prom/process | python3 -m json.tool

This is equivalent to clicking the button. Useful for scripting or triggering from another system.

Configuration Reference

All behavior is controlled through environment variables. Set them in the cron entry or in a shell profile file:

VariableDefaultDescription
SCANFOR_PROM_FILE_PATHproject/test_data/Absolute path to the folder containing .prom files. On the server: /home/plmon/status/scanfor
SCANFOR_PROM_POLL_SECONDS600Poll interval for the built-in watcher (if enabled). Not used by cron.
SCANFOR_ENABLE_PROM_WATCHERfalseSet to true to enable the built-in async watcher instead of cron. Pick one approach, not both.

Troubleshooting

SymptomCauseFix
No log output after 10 minutes Cron not running, or cron daemon not started Check: crontab -l to confirm entry exists. On macOS: cron is enabled by default. On Linux: systemctl status cron or systemctl status crond
ModuleNotFoundError: No module named 'backend' Python can't find the project modules вЂ” working directory is wrong Make sure the cron entry has cd /opt/scanfor-red && before the python command
prom path not found SCANFOR_PROM_FILE_PATH points to a path that doesn't exist Verify: ls /home/plmon/status/scanfor. Check spelling and that the directory exists on this server.
Log shows "skipped" every run .prom files not being updated by Scanfor, or path points to wrong directory Check that Scanfor is running on the server. Verify file modification times: ls -lt /home/plmon/status/scanfor/*.prom
Dashboard not updating after cron runs Data IS in the DB but browser is showing cached data Click Refresh Data in the dashboard, or hard-reload (Cmd+Shift+R / Ctrl+Shift+R)
OperationalError: database is locked Two processes writing to SQLite simultaneously Rare with 10-minute intervals. If cron takes more than 10 min, prevent overlap: add flock -n /tmp/scanfor_cron.lock before the python command

Removing the Cron Job

# Remove all cron entries for your user (CAREFUL вЂ” removes everything):
crontab -r

# Or edit and remove just the Scanfor line:
crontab -e
