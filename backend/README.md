# Scanfor Red ‚Äî Backend API (Phase 2)

FastAPI backend that serves alert triage data to the dashboard frontend.

## Quick Start

From the **project root** (`Scanfor_Red_Email_Alerts_Dashboard/`):

```bash
# Install dependencies (one-time)
pip3 install -r requirements.txt

# Start the API server
python3 -m uvicorn backend.main:app --reload --port 9000
```

The API runs at `http://localhost:9000`. Auto-reload is on ‚Äî save any Python file and it restarts.

## Endpoints

| Method | Path | Description |
|---|---|---|
| GET | `/api/health` | Service health check |
| GET | `/api/summary` | Summary card counts for the dashboard |
| GET | `/api/alert-batches/latest` | Latest alert batch metadata |
| GET | `/api/alerts` | Classified alert events. Defaults to `signal_type=actionable`; pass `?include=noise`, `?include=suppressed`, or `?include=all` to see other classifications |
| GET | `/api/known-issues` | Known issue catalog |

Interactive docs (auto-generated by FastAPI): http://localhost:9000/docs

## Where Mock Data Lives

| File | Contents |
|---|---|
| `backend/data/mock_alerts.py` | `ALERT_BATCH`, `RAW_ALERTS` (active), `RESOLVED_ALERTS` |
| `backend/data/mock_known_issues.py` | `KNOWN_ISSUES` ‚Äî the 5 catalog records |

Edit these files to change what the dashboard displays. The classifier runs at request
time, so changes to `normal_count_max` thresholds immediately affect worsening detection.

## How Classification Works

1. Each raw alert is passed to `classifier.classify_alert()`
2. The classifier calls `find_matching_known_issue()` which checks host scope, log scope,
   and error type against every known issue record
3. If no match ‚Üí `new`; if match and within thresholds ‚Üí `known`; if match but
   count or growth exceeds `normal_count_max` ‚Üí `worsening`
4. Resolved alerts are passed through as-is (category is set to `resolved`)

## Actionable / Noise / Suppressed Classification

Separately from `category` above, every alert also gets a `signal_type` ‚Äî this only
controls what shows up in the default dashboard/API view, never what's stored:

- `classifier.classify_alert_signal(color, known_error, actionable_colors, suppress_known_errors)`
- Driven by `SCANFOR_ACTIONABLE_COLORS` (default `red,yellow`) and
  `SCANFOR_SUPPRESS_KNOWN_ERRORS` (default `true`) in `backend/services/config.py`
- Every parsed row is stored regardless of its `signal_type` ‚Äî nothing is filtered
  before persistence, so escalation/resolution history stays intact even for rows
  that start out as noise or suppressed

## Running Tests

```bash
pip3 install -r requirements-dev.txt
python3 -m pytest tests/ -v
```

Tests use an isolated temp SQLite file (set up in `tests/conftest.py`) ‚Äî they never
touch your local `backend/database/scanfor_red.db`.
